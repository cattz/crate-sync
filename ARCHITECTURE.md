# Crate Sync вҖ” Architecture

## Stack

| Component | Choice | Rationale |
|-----------|--------|-----------|
| **Language** | TypeScript (ESM) | Single language, strong typing, good ecosystem |
| **Runtime** | Node.js | Mature, good library support for all integrations |
| **Package Manager** | pnpm | Fast, disk-efficient |
| **CLI Framework** | Commander.js | Simplest, most proven (~308M weekly downloads) |
| **Database** | SQLite via better-sqlite3 | Synchronous, fast, zero-config |
| **ORM** | Drizzle ORM + drizzle-kit | Schema-first, typed queries, built-in migrations |
| **Spotify API** | spotify-web-api-ts or @spotify/web-api-ts-sdk | Official/typed Spotify client |
| **Soulseek** | TBD вҖ” see below | Critical dependency, limited options |
| **Lexicon DJ** | Custom HTTP client | REST API at `localhost:48624/v1` |
| **Audio Metadata** | music-metadata | Read MP3 ID3 / FLAC Vorbis tags for validation |
| **Fuzzy Matching** | fuse.js or custom | Weighted scoring with configurable thresholds |
| **Testing** | vitest | Fast, native TypeScript/ESM support |
| **Build** | tsup | Simple, fast bundling for CLI |

### Soulseek Client вҖ” Options

This is the riskiest dependency. Options in order of preference:

1. **slskd REST API** вҖ” Run [slskd](https://github.com/slskd/slskd) as a local server, talk to it via HTTP. Most reliable, actively maintained, decouples protocol complexity. Adds a runtime dependency (Docker or binary).
2. **andrade-soulseek-downloader** вҖ” npm package, single maintainer. Has built-in rate limiting. Risk: could go unmaintained.
3. **soulseek-ts** вҖ” TypeScript native, but stale (~2 years no updates).
4. **Port from Python** вҖ” Use aioslsk logic as reference to build our own. High effort.

**Decision**: Use **slskd REST API**. Searches are rate-limited (configurable delay). Downloads are concurrent (no rate limit).

## Key Design Decisions

### 1. Commands are thin, services are fat

Commands parse args and call services. All business logic lives in services. This keeps commands testable and allows reuse when we add TUI/Web later.

```
Command (parse args) вҶ’ Service (business logic) вҶ’ DB / External API
```

### 2. Matching is a first-class abstraction

```typescript
interface MatchStrategy {
  match(source: TrackInfo, candidates: TrackInfo[]): MatchResult[];
}

interface MatchResult {
  candidate: TrackInfo;
  score: number;         // 0.0 - 1.0
  confidence: 'high' | 'review' | 'low';
  method: string;        // 'isrc' | 'fuzzy' | etc.
}
```

Strategies are composable. The `CompositeStrategy` tries ISRC first (exact), then falls back to fuzzy. Each context (dedup, Lexicon sync, Soulseek search) can configure its own thresholds.

### 3. Three-phase pipeline

The sync pipeline separates concerns to avoid blocking the user:

```
Phase 1: MATCH (batch)     вҖ” All tracks matched against Lexicon, no user input needed
Phase 2: REVIEW (interactive) вҖ” User confirms/rejects all uncertain matches at once
Phase 3: DOWNLOAD (autonomous) вҖ” Searches + downloads run unattended
```

State is persisted between phases so the pipeline can be resumed.

### 4. Lexicon integration via REST API

Lexicon exposes a local REST API at `http://localhost:48624/v1`. Key quirks from existing implementations:

- **Playlist update is REPLACE, not append** вҖ” always fetch current trackIds, merge, then send back
- **IDs can be int or string** вҖ” normalize to string
- **Response wrapping varies** вҖ” handle `{data: {...}}`, `{tracks: [...]}`, and bare arrays
- **Search uses bracket filter syntax** вҖ” `filter[artist]=X&filter[title]=Y`

### 5. Soulseek rate limiting

Searches are rate-limited (configurable delay between searches) to avoid bans. Downloads are not rate-limited вҖ” they can run concurrently.

### 6. Downloaded files go to Lexicon incoming folder

After download completes:
1. **Validate tags** вҖ” Read MP3 ID3 / FLAC Vorbis tags and verify artist+title match the expected track
2. **Rename** вҖ” `<Artist> - <Title>.<ext>` (sanitized for filesystem)
3. **Move** вҖ” Place in `<downloadRoot>/<playlistName>/`
4. **Import** вҖ” Lexicon watches the download root and auto-imports new files
5. **Playlist** вҖ” Re-match against Lexicon, add to playlist preserving Spotify track order

## Configuration

Single config file at `~/.config/crate-sync/config.json`:

```json
{
  "spotify": {
    "clientId": "...",
    "clientSecret": "...",
    "redirectUri": "http://localhost:8888/callback"
  },
  "lexicon": {
    "url": "http://localhost:48624",
    "downloadRoot": "/path/to/music/downloads"
  },
  "soulseek": {
    "slskdUrl": "http://localhost:5030",
    "slskdApiKey": "...",
    "searchDelayMs": 5000
  },
  "matching": {
    "autoAcceptThreshold": 0.9,
    "reviewThreshold": 0.7
  },
  "download": {
    "formats": ["flac", "mp3"],
    "minBitrate": 320
  }
}
```

CLI flags override config file values.

## Database Schema (Drizzle)

See `FEATURES.md` for entity descriptions. Core tables:

- `playlists` вҖ” Cached Spotify playlists
- `tracks` вҖ” Cached Spotify tracks
- `playlist_tracks` вҖ” Junction table with position
- `lexicon_tracks` вҖ” Cached Lexicon library tracks
- `matches` вҖ” Match registry (source, target, score, status)
- `downloads` вҖ” Download state machine (pending вҶ’ searching вҶ’ downloading вҶ’ validating вҶ’ done/failed)
- `sync_log` вҖ” Audit trail
